# App Auto Update Design

## Goal

Add app-level auto update support for Skiller Desktop. Skiller should check for app updates in the background, download available updates automatically, then show an `Update` button beside the `Skiller` heading in the left panel once the update is ready. Clicking the button installs the downloaded update and restarts the app.

This feature is separate from skill updates. The existing `Updates` page continues to manage installed skill updates.

## Scope

- Support packaged macOS builds.
- Support Linux AppImage builds when AppImage update requirements are available.
- Hide app update UI for development builds, Linux `.deb` installs, and unsupported packaging.
- Publish the updater metadata required by Electron Builder.
- Keep user-visible app update UI in the left panel heading.

Windows update support is out of scope for this version.

## Architecture

Add an app update service under `apps/desktop/src/main`. The service owns `electron-updater`, starts after `app.whenReady()`, checks GitHub Releases on startup and periodically in the background, downloads updates automatically, and exposes normalized state to the renderer. The app update check interval is owned by this service and stays separate from the existing skill update schedule.

Normalized states:

- `idle`
- `checking`
- `downloading`
- `ready`
- `not-available`
- `unsupported`
- `error`

The state includes version, release name, release date, progress, and error message when available. The service logs failures in the main process and emits state changes to renderer windows.

Expose app update IPC through preload:

- `getAppUpdateState()`
- `checkAppUpdate()`
- `installAppUpdate()`
- `onAppUpdateState(callback)`

`installAppUpdate()` calls `quitAndInstall()` only after the service reaches `ready`.

## Release Metadata

Add `electron-updater` as an `@skiller/desktop` app dependency.

Add Electron Builder GitHub publish config for `gannonh/skiller`. Build output must include updater metadata:

- `latest-mac.yml`
- `latest-linux.yml`

Update `.github/workflows/desktop-release.yml` so GitHub releases upload the metadata files with the existing DMG, zip, AppImage, and deb assets.

## UI

`apps/desktop/src/renderer/App.tsx` owns the left panel heading. It subscribes to app update state through the preload API.

The header keeps the `Skiller` title and `Agent skill manager` subtitle. The right side contains the theme toggle and, when state is `ready`, a compact `Update` button. The button includes accessible text such as `Install app update 0.2.2` when the version is known.

The button stays hidden for `idle`, `checking`, `downloading`, `not-available`, `unsupported`, and `error`.

The skill `Updates` page keeps its current route, label, and behavior.

## Platform Behavior

macOS uses the signed and notarized zip metadata generated by Electron Builder.

Linux auto-update applies only to AppImage runs where updater requirements are present. `.deb` installs report `unsupported` and hide the header button.

Development runs report `unsupported`. Tests can inject explicit dev updater config.

## Error Handling

Network, metadata, signature, download, and unsupported-platform failures produce explicit states. The app logs details in the main process and exposes concise renderer state.

The implementation should not open browser downloads as a fallback. It should not route app update errors through the skill update page.

## Testing

Unit tests cover:

- updater events mapped to normalized states
- unsupported platform state
- `installAppUpdate()` guarded by `ready`
- preload app-update API exposure
- IPC registration and delegation

Renderer tests cover:

- sidebar button hidden before update readiness
- sidebar button shown when an update is ready
- button click invokes `installAppUpdate()`
- skill `Updates` page behavior remains independent

Release validation covers:

- Electron Builder output includes `latest-mac.yml` and `latest-linux.yml`
- GitHub release workflow uploads updater metadata
- `pnpm check` passes

## Success Criteria

- A packaged supported build checks for app updates in the background.
- A downloaded app update makes the left-panel `Update` button appear beside `Skiller`.
- Clicking `Update` restarts into the downloaded version.
- Unsupported builds hide the button and expose an explicit unsupported state.
- Skill update workflows remain unchanged.
